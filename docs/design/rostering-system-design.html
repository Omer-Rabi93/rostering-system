Rostering System — Technical Design

A multi-company, planner-facing web application that manages security workers and their contracts, and auto-generates a monthly roster over three fixed daily shifts (A 00:00–08:00, B 08:00–16:00, C 16:00–00:00) independently per company. The design is a conventional three-tier stack — a React + TypeScript SPA, a Node.js/Express REST API, and PostgreSQL — with the scheduling engine as a CP-SAT constraint-optimization model (Google OR-Tools) run in a small Python sidecar. All hard rules (availability, role match, the 2-shifts-per-calendar-day cap) are enforced server-side by one shared validator used by both auto-generation and manual edits, so a contract violation cannot be persisted through any path. Soft rules (min/max monthly hours) surface as alerts the planner must explicitly acknowledge before a roster can be saved. There are no worker logins — workers get a read-only, printable schedule page — and, as of this revision, no planner logins either (see Security & Scope Boundaries).

Design revision (this document): rewritten to reflect the full body of work since "Availability v2" (2026-07-17) — every company now has its own independent worker pool, staffing requirements, and roster (previously one global candidate pool shared across all companies); worker/availability CSV imports are company-scoped, tracked as first-class ImportTask rows with cancel-and-replace semantics instead of a single ad hoc pg-boss job; the background-job layer grew a third queue and per-company partitioning; and the whole Docker/nginx/scaling deployment layer — entirely undocumented in the previous revision — is now covered. Sections below are labeled shipped, in progress, or deferred against what's actually merged to main as of this writing; see Known Limitations & Roadmap for the full picture.

Multi-Tenancy & Company Scoping

Every company is a fully independent scheduling unit: its own workers, its own staffing requirements, its own roster per month. Nothing in the scheduling engine or candidate-selection logic ever mixes workers or requirements across companies — StaffingRequirement and Roster both carry a required companyId and are uniquely keyed on (companyId, ...), and RosterGenerationService.generate() filters every input query (active workers, requirements, availability) by companyId before ever building the CP-SAT problem.

This is not a server-enforced authorization boundary — see Security & Scope Boundaries — it's a data-partitioning and UX concept. On the frontend, an "active company" is chosen once (create-or-pick, gated in front of every authenticated screen — ActiveCompanyGate.tsx) and persisted to localStorage; every subsequent page/query/mutation threads that id through. A company switcher in the top bar changes it without navigating away.

The active-company gate is a client-side UX convenience, not an authorization boundary. This is a deliberate, documented scope limitation (see Security & Scope Boundaries), not an oversight.

Projected cost per worker = assigned shifts × 8 × hourly rate; the roster total is the sum.

Design decision: roster lifecycle is draft → published; regenerating or re-editing an already-published month reopens it as a draft, and publishing again re-runs the full alert/acknowledgment gate.

Scheduling Engine Flow

The engine is a CP-SAT constraint-optimization model (Google OR-Tools), built and solved once per (company, month) generation job in a small Python sidecar. The whole month is optimized in one model rather than filled slot by slot, using only that company's own active workers and requirements:

  • Decision variables. Booleans x[w,d,s] ∈ {0,1} — worker w assigned to day d, shift s. A variable is only created where worker w has a WorkerAvailability row for the exact calendar date d whose shift subset contains s. A worker with no availability rows in the month contributes no variables at all.
  • Hard constraints. (a) Role coverage per slot: for each day × shift × role, the sum of x over eligible workers of that role plus a shortfall slack equals the required headcount; (b) ≤ 2 shifts per worker per calendar day; (c) max monthly hours: 8·Σx ≤ maxMonthlyHours per worker.
  • Soft constraints via slack variables. Coverage shortfall slacks (each positive value becomes an unfillable_slot alert) and per-worker min-hours shortfall slacks (each positive value becomes a min_hours_shortfall alert).
  • Objective (lexicographic via weights). Minimize coverage shortfalls first, then min-hours shortfalls, then a fairness term minimizing the spread between the most- and least-loaded workers.
  • Determinism & degradation. Fixed random_seed, num_search_workers = 1, and a ~30 s solver time limit. The model is built so a feasible solution should always exist (slacks absorb any shortage), but on a large enough problem the solver can still exhaust its time budget before recording any solution — the solver now checks its own status and fails loudly with a clear error in that case, rather than crashing with an opaque exception (a real bug found by load-testing at ~1,000 workers, fixed).

Manual edits never touch the solver. Every move/add/remove on the calendar goes through the server-side TypeScript RosterValidator: hard violations are rejected outright; min/max-hours breaches return a warning the UI turns into a confirm dialog.

Design decision: CP-SAT is chosen over a greedy assignment pass because it finds globally optimal rosters and handles the coverage / min-hours / fairness interplay in one model instead of heuristic tie-breaks.
Known scaling limit: generation currently always solves and persists the entire month from scratch, every time — there's no way to regenerate just a range of dates while leaving the rest of the month's assignments untouched. See Known Limitations & Roadmap.

Background Jobs, Queues & Import Tasks

Heavy or long-running work never runs inside an HTTP request. The API enqueues a pg-boss job and returns 202 {jobId} immediately; a separate worker process picks jobs off the queue, executes them, and (for CSV imports) also maintains a first-class ImportTask row the UI polls independently of the raw pg-boss job record.

QueueConcurrencyPartitioning
csv-importCSV_IMPORT_CONCURRENCY, default 8 — I/O-bound, safe to raise (bounded by the Prisma connection pool) singletonKey = "companyId:WORKER_SYNC"
availability-importAVAILABILITY_IMPORT_CONCURRENCY, default 8, same reasoningsingletonKey = "companyId:AVAILABILITY_SYNC"
roster-generationROSTER_GENERATION_CONCURRENCY, default 2 — CP-SAT is CPU-bound and deliberately single-threaded per solve (for determinism), so this should track available CPU cores per worker replica, not be raised arbitrarily singletonKey = "companyId:month"

All three queues use pg-boss's stately policy, which requires an explicit singletonKey to partition uniqueness per company — omitting it does not disable the uniqueness constraint, it collapses every job on the queue onto one shared implicit key, making the whole queue globally single-flight across every company instead of per-company (a regression found and reverted during development; the key is required, not an optimization).

Cancel-and-replace: at most one import in flight per company

A worker-CSV or availability-CSV upload is tracked as an ImportTask row (PENDING → PROCESSING → COMPLETED/FAILED/CANCELLED), independent of pg-boss's own generic job bookkeeping. Uploading a new file while one is still processing for the same company+kind cancels the in-progress one and proceeds with the new one:

  1. The route marks any existing non-terminal task for that company+kind CANCELLED and best-effort cancels its pg-boss job, then creates a fresh PENDING task.
  2. boss.cancel() reliably stops a job that hasn't started yet, but cannot forcibly interrupt code already executing inside a running handler — so the row-processing loop itself re-checks its own task's status every ~50 rows and stops (without ever reaching COMPLETED) if it's been superseded.
  3. A worker's lastImportTaskId is stamped in one bulk write, only once the task is confirmed COMPLETED — never per-row mid-loop. An earlier version stamped it optimistically per row, which allowed a worker already correctly stamped by an earlier completed sync to have that valid stamp overwritten and lost by a later task that touched their row before itself getting cancelled — a real bug, found at scale (5,000+ workers mis-stamped in a load test) and fixed.
  4. Two independently-raced resources — the import_tasks DB row (guarded by a partial unique index, (companyId, kind) where status is non-terminal) and the pg-boss job row (guarded by singletonKey) — mean a single retry attempt is not enough under genuine concurrent load; the route retries the whole cancel-then-enqueue sequence, not just one half of it, up to a bounded count.

Under a sustained, realistic upload cadence (tested: 60 uploads over 60 seconds, each replacing 50–100% of the file's contents) this converges cleanly to exactly one COMPLETED task with zero data corruption. Under a more extreme, less realistic pattern (10 literally simultaneous requests within the same sub-second window) the retry can still occasionally fail to converge to a single winner — documented as a known limitation (no crash, no data corruption, just an occasional clean "nobody won, try again"), not chased further since it isn't the scenario the system needs to handle. A fully airtight fix would need a Postgres advisory lock serializing the whole sequence.

Roster-eligibility gate

A worker dropped from a new CSV upload stays ACTIVE (no automatic deactivation — an earlier, removed "sync sweep" behavior silently deactivated workers, and did so globally across every company, not just the uploading one, since company resolution used to happen per-row rather than per-upload). Instead, they become ineligible for roster generation until they reappear in a completed sync: RosterGenerationService only considers active workers whose lastImportTaskId matches the company's latest completed WORKER_SYNC task, or who have never been touched by a CSV sync at all (so a company that manages workers entirely by hand can still generate a roster).

The queue is DB-backed: pg-boss, claiming jobs with SELECT … FOR UPDATE SKIP LOCKED, so no infrastructure beyond Postgres is needed. It also supplies the cron scheduler (06:00 on the 25th of every month, auto-generates next month's draft — never publishes) and the singleton-key mechanism described above.

Design decision: job execution is asynchronous but results are delivered by UI polling of the job/task record — no WebSockets or push channels.

  @@map("shift_workers")
}

model Alert {                                       // table: alerts
  id             Int       @id @default(autoincrement())
  rosterId       Int
  roster         Roster    @relation(fields: [rosterId], references: [id], onDelete: Cascade)
  type           AlertType
  detail         Json      @db.JsonB
  acknowledged   Boolean   @default(false)
  acknowledgedAt DateTime?
  @@index([rosterId])
  @@map("alerts")
}
Design decision: the shifts–workers connection is normalized into a shifts table (one row per roster + date + shift type) plus a shift_workers junction that snapshots the worker's role at assignment time, so a historical roster stays accurate even if the worker is later promoted or reassigned.

Schema optimization notes

  • Native Postgres enums everywhere a fixed set of values applies — invalid values are rejected by the database itself.
  • Every FK used in joins is indexed: companyId on Worker/StaffingRequirement/Roster, plus the ImportTask composite lookups above.
  • Partial indexes: ON workers (id) WHERE status = 'ACTIVE', and the import_tasks "at most one non-terminal task per company+kind" uniqueness constraint described in Background Jobs.
  • Cascade rules: ON DELETE CASCADE down roster → shifts → shift_workers and roster → alerts; ON DELETE RESTRICT from companies → workers, companies → staffing requirements/rosters, and workers → shift_workers (no employer, requirement setting, or shift history is ever silently orphaned); ON DELETE SET NULL from workers → their last import task.
  • No stored derived values: hours and cost are always computed at read time.

pg-boss schema: pg-boss creates and migrates its own pgboss schema on first start; it lives outside Prisma.

Core Algorithms & Validation

RosterValidator runs five hard rules on every manual edit, in this order: workerIsActive, roleMatchesSlot, withinAvailability, noDuplicateSlot, maxTwoShiftsPerDay. Any violation returns 422 and the edit is never persisted. Soft rules (min/max monthly hours) return 409 with a warning; the client resubmits the identical request with ?confirm=true to proceed anyway, and alerts are recomputed so the save gate always reflects the current grid.

Israeli national ID validation uses the standard weighted-digit checksum (alternating ×1/×2 with digit-sum reduction), applied uniformly at every write boundary (manual worker create/edit, CSV import) via a shared Zod schema — never duplicated logic.

CSV Schema & Import Pipeline

shipped Two independent CSV pipelines today; a merge into one combined upload is in progress — see Roadmap.

Worker CSV — 7 columns

national_id,name,role,status,hourly_cost_ils,min_monthly_hours,max_monthly_hours

Scoped to one company at upload time (a required companyId form field) — not resolved per row. (History: originally 18 columns including 10 weekly-availability columns and a per-row company_name column; the availability columns were dropped when availability became date-specific, and company_name was dropped when upload became company-scoped — resolving per row was a real bug, since it let one upload silently touch/deactivate workers at companies never mentioned in the file.) Matching is by national_id (upsert); a row whose ID already belongs to a different company is a per-row error, never a silent reassignment. No automatic deactivation — see the roster-eligibility gate in Background Jobs for how "absent from the latest sync" is actually handled.

Availability CSV — national_id + one column per calendar day

national_id,d01,d02,...,d<N>   // N = the target month's actual day count

Also scoped to one company at upload time. Cell = empty (no row — currently means unavailable that date, see the semantics-inversion note in Roadmap) or a canonical ordered shift-subset string (A, AB, ABC, …). File-level errors (bad header, wrong row shape) abort the whole import; a bad cell inside an otherwise well-formed row fails only that row. Formula-injection guarding applied uniformly to every cell.

Both pipelines: MAX_ROWS = 10,000; imports run as a pg-boss job (202 + jobId), tracked as an ImportTask with the cancel-and-replace semantics described in Background Jobs; each row runs in its own transaction so one bad row never aborts the batch, and the job result includes a per-row error report.

Security & Scope Boundaries

This section states plainly, with no hedging, what this system deliberately does not do — verified directly against the code, not inferred:

  • No authentication anywhere. No login, session, JWT, API key, or any other credential mechanism exists on any route. Every endpoint except the public schedule page is reachable by anyone who can reach the server, with no identity check. The "active company" concept is a client-side UX convenience only — nothing server-side ties a request to "the company this caller is allowed to act as," and nothing prevents a client from passing any companyId it likes.
  • No role-based permissions or audit trail. Single planner/HR user model — the schema leaves room to add both later, but neither exists today. Worker.role is a job title (General Guard / Supervisor / Screener), not an application permission.
  • No CORS configuration. By design, not oversight — the containerized deployment serves the SPA and proxies /api from the same origin via nginx, so no cross-origin request is ever expected. Running the API standalone against a different-origin frontend would require adding CORS handling.
  • Rate limiting is minimal and narrow. Only /api/schedule/* (the one intentionally public, unauthenticated route) is rate-limited — 30 requests/minute per IP, both at the application layer (express-rate-limit) and again at the nginx edge (defense-in-depth). Every other route has no rate limiting at all.
  • No WebSockets or push channels. All async results (CSV import progress, roster generation) are delivered by client-side polling of the job/task record.
  • The public schedule link is the one deliberate exception to "no authentication" — access control there is a per-worker, non-guessable, rotatable share token (UUID), not a login. It exposes only that one worker's own name and published shifts, never national ID, hourly rate, or any other worker's data.
Practical implication: this system is appropriate for a trusted, single-operator deployment (one planner or HR team operating it directly, on a network they control) — not for exposing directly to the public internet or to multiple mutually untrusting parties without adding an authentication/authorization layer in front of it first.

Scaling

Both api and worker have no container_name and no host-port binding, so both scale via docker compose up --scale api=N --scale worker=N with no config change needed. Scaling the worker is safe specifically because the "at-most-one-non-terminal-task-per-company" invariant is enforced at the database level (Background Jobs's cancel-and-replace sequence + partial unique index) before a job is ever handed to pg-boss — adding more worker replicas just adds more concurrent claimers of an already-correct, already-deduplicated pool of job rows; it can't create a race that the DB layer doesn't already prevent.

The real ceiling on replica count is Postgres connections, not CPU: each process holds two separate, independently-sized connection pools — Prisma's own (DB_POOL_SIZE_PRISMA, default 10) and pg-boss's (DB_POOL_SIZE_BOSS, default 10) — so a worker replica alone can hold up to ~20 connections. At Postgres's default max_connections = 100, that's a practical ceiling around 4–5 total replicas before either lowering the per-instance pool sizes or introducing PgBouncer becomes necessary.

ROSTER_GENERATION_CONCURRENCY should be tuned to available CPU cores per worker replica, not raised freely like the two CSV queues — CP-SAT is deliberately single-threaded per solve (for determinism), so throughput scales with cores, and setting concurrency above the actual core count just makes every concurrent solve slower via OS scheduling contention, not faster overall.

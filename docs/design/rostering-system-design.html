Rostering System — Technical Design

A single-site, planner-facing web application that manages security workers and their contracts, and auto-generates a monthly roster over three fixed daily shifts (A 00:00–08:00, B 08:00–16:00, C 16:00–00:00). The design is a conventional three-tier stack — a React + TypeScript SPA, a Node.js/Express REST API, and PostgreSQL — with the scheduling engine as a CP-SAT constraint-optimization model (Google OR-Tools) run in a small Python sidecar invoked by the roster-generation worker, while all validation and orchestration stay in TypeScript. All hard rules (availability, the 2-shifts-per-calendar-day cap) are enforced server-side by one shared validator used by both auto-generation and manual edits, so a contract violation cannot be persisted through any path. Soft rules (min/max monthly hours) surface as alerts that the planner must explicitly acknowledge before a roster can be saved. There are no worker logins: workers get a read-only, printable schedule page.

Components & Responsibilities

ComponentOwns
Web UI (SPA) Worker/contract CRUD screens, staffing-requirements settings, monthly calendar grid with drag/click editing, the pre-save alert checklist, and the cost dashboard. Client-side checks are hints only (e.g. greying out unavailable workers in the edit dialog) — never the source of truth.
REST API All business validation, server-side and authoritative: Israeli-ID checksum, contract bounds, and the shared RosterValidator that blocks hard-rule violations (HTTP 422) and returns soft-rule warnings requiring a confirmation flag. Also enforces the save gate: a roster with unacknowledged alerts cannot be saved.
Scheduling Engine A CP-SAT solver sidecar: a small Python script built on Google OR-Tools CP-SAT. The Node worker spawns it with a JSON problem file (month, workers + contracts, staffing requirements) and reads back a JSON solution (assignments + slack-derived alerts). Still pure and deterministic — fixed seed, single search worker — with no side effects beyond stdin/stdout. The RosterValidator remains in TypeScript and still gates all manual edits, so nothing invalid is ever persisted through any path.
CSV Import/Export One combined workers+contracts schema; export is re-importable unmodified. Availability is encoded as 21 boolean columns (avail_sun_a … avail_sat_c). Import validates per row, upserts by Israeli ID, and returns a per-row error report — one bad row never aborts the batch. Import is a full sync: existing workers whose ID is absent from the file are set Inactive (never deleted — contract and shift history kept).
Background Workers A worker process consuming a job queue: CSV imports and roster generation run asynchronously instead of inside the HTTP request, and a scheduler triggers recurring (cron) jobs. Workers reuse the same engine, validator, and CSV modules as the API — no duplicated logic.
Relational DB PostgreSQL. Uniqueness and referential integrity at the schema level, including the shift_workers composite primary key that makes duplicate slot assignments impossible. Scales comfortably with significant workforce growth; the engine's working set for one month fits in memory.
Design decision: the worker schedule view is a read-only page served at a non-guessable per-worker token URL (plus a print stylesheet), so no worker accounts are needed.
Design decision: single planner/HR user model — no role-based permissions or audit trail in this version; the schema leaves room to add both later.

Scheduling Engine Flow

The engine is a CP-SAT constraint-optimization model (Google OR-Tools), built and solved per generation job in a small Python sidecar. The whole month is optimized in one model rather than filled slot by slot:

  • Decision variables. Booleans x[w,d,s] ∈ {0,1} — worker w assigned to day d, shift s. A variable is only created where the worker's contract availability allows that weekday + shift, which prunes the model before solving even starts.
  • Hard constraints. (a) Role coverage per slot: for each day × shift × role, the sum of x over workers of that role plus a shortfall slack equals the required headcount; (b) ≤ 2 shifts per worker per calendar day; (c) max monthly hours: 8·Σx ≤ maxMonthlyHours per worker.
  • Soft constraints via slack variables. Coverage shortfall slacks — every positive value in the solution becomes an unfillable_slot alert — and per-worker min-hours shortfall slacks (8·Σx + shortfall ≥ minMonthlyHours), each positive value becoming a min_hours_shortfall alert.
  • Objective (lexicographic via weights). Minimize coverage shortfalls first (large weight), then min-hours shortfalls, then a fairness term: minimize the spread between the most- and least-loaded workers relative to their contracted hours — the even-distribution preference expressed directly in the objective.
  • Determinism & degradation. Fixed random_seed, num_search_workers = 1, and a solver time limit (e.g. 30 s) with the best feasible solution accepted. Infeasibility is impossible by construction — the slacks absorb any shortage — so the model always returns a roster plus alerts, never an error.

Manual edits never touch the solver. Every move/add/remove on the calendar goes through the server-side TypeScript RosterValidator: hard violations (third shift in a day, outside availability) are rejected outright; min/max-hours breaches return a warning the UI turns into a confirm dialog, and alerts are recomputed so the save gate always reflects the current grid.

Design decision: CP-SAT is chosen over a greedy assignment pass because it finds globally optimal rosters and handles the coverage / min-hours / fairness interplay in one model instead of heuristic tie-breaks; the trade-off — a Python sidecar and solver runtime — is absorbed by the async job worker, where a few seconds of solving is invisible to the API.
Design decision: back-to-back Shift C → next-day Shift A is allowed — the brief mandates only the per-calendar-day cap and no further labour-law simulation.

Background Jobs & Workers

Heavy or long-running work never runs inside an HTTP request. The API enqueues a Job row and returns immediately with the job id; a separate worker process picks jobs off the queue, executes them, and writes progress and results back to the job record, which the UI polls.

  • CSV import worker. Parses and validates the uploaded file row by row using the shared CSV module, upserts valid rows by national ID, then runs the sync sweep: every existing worker whose national ID appears nowhere in the file (valid or failed row) is set Inactive. Stores the per-row error report plus the deactivation list as the job result — a large file can never time out the request, and one bad row never aborts the batch.
  • Roster generation worker. Runs the CP-SAT sidecar for the requested month: it builds the problem JSON, spawns python3 solver/solve_roster.py, passes the problem on stdin, parses the solution JSON (assignments + alerts), and persists the draft roster — the month's shifts rows plus their shift_workers rows — and alerts inside one transaction as the job result. The API stays responsive while the solver runs, and generation for a month can't run twice concurrently (one in-flight roster_generation job per month).
  • Scheduler (cron). A recurring job runs near the end of each month (e.g. the 25th) to auto-generate next month's draft roster with its alerts, so planners start from a prepared draft instead of a blank month. The draft still goes through the normal review, acknowledgment, and publish gate — cron never publishes.

The queue is DB-backed: pg-boss, a PostgreSQL-based job queue that claims jobs with SELECT … FOR UPDATE SKIP LOCKED, so no infrastructure beyond Postgres is needed. pg-boss also supplies the cron scheduler and per-key singleton jobs (used for the one-in-flight generation-per-month rule); swapping in a dedicated queue (e.g. Redis) later is an implementation detail behind the same job interface. Queue names, retry policy, the cron entry, and the polling endpoint shape are specified in CSV Schema & Jobs Detail.

Design decision: job execution is asynchronous but results are delivered by UI polling of the job record — no WebSockets or push channels in this version.

Database Schema

The full Prisma schema (PostgreSQL). Every business-critical uniqueness rule is a database constraint, not just application logic:

enum Role         { GENERAL_GUARD  SUPERVISOR  SCREENER }
enum ShiftType    { A  B  C }                       // A 00–08, B 08–16, C 16–24
enum WorkerStatus { ACTIVE  INACTIVE }
enum RosterStatus { DRAFT  PUBLISHED }
enum AlertType    { UNFILLABLE_SLOT  MIN_HOURS_SHORTFALL }

model Company {                                     // table: companies
  id        Int      @id @default(autoincrement())
  name      String   @db.VarChar(120)               // unique case-insensitively: raw-SQL migration
                                                    // CREATE UNIQUE INDEX ON companies (lower(name))
  createdAt DateTime @default(now())
  workers   Worker[]
  @@map("companies")
}

model Worker {                                      // table: workers
  id          Int              @id @default(autoincrement())
  nationalId  String           @unique @db.Char(9) // Israeli ID; checksum checked in Zod
  name        String           @db.VarChar(120)
  companyId   Int                                   // ← employer, required
  company     Company          @relation(fields: [companyId], references: [id], onDelete: Restrict)
  role        Role
  status      WorkerStatus     @default(ACTIVE)
  shareToken  String           @unique @default(uuid())  // public schedule URL token
  createdAt   DateTime         @default(now())
  updatedAt   DateTime         @updatedAt
  contract    Contract?
  shiftWorkers ShiftWorker[]
  @@index([companyId])                              // company filters/joins
  @@index([status])                                 // engine loads ACTIVE workers (see partial index note)
  @@map("workers")
}

model Contract {                                    // table: contracts
  id              Int      @id @default(autoincrement())
  workerId        Int      @unique                  // ← one contract per worker
  worker          Worker   @relation(fields: [workerId], references: [id], onDelete: Cascade)
  hourlyCostIls   Decimal  @db.Decimal(8, 2)        // NOT NULL, ≥ 0 (Zod)
  minMonthlyHours Int                               // NOT NULL, 0 ≤ min ≤ max (Zod)
  maxMonthlyHours Int
  availability    Json     @db.JsonB                // 7×3 boolean matrix, Zod-validated on write
  updatedAt       DateTime @updatedAt
  @@map("contracts")
}

model StaffingRequirement {                         // table: staffing_requirements
  id            Int       @id @default(autoincrement())
  role          Role
  shift         ShiftType
  requiredCount Int                                 // NOT NULL, ≥ 0
  @@unique([role, shift])                           // ← one row per role × shift (≤ 9 rows)
  @@map("staffing_requirements")
}

model Roster {                                      // table: rosters
  id          Int          @id @default(autoincrement())
  month       String       @unique @db.Char(7)     // ← "YYYY-MM"; one roster per month
  status      RosterStatus @default(DRAFT)
  generatedAt DateTime?
  publishedAt DateTime?
  shifts      Shift[]
  alerts      Alert[]
  @@map("rosters")
}

model Shift {                                       // table: shifts
  id        Int       @id @default(autoincrement())
  rosterId  Int
  roster    Roster    @relation(fields: [rosterId], references: [id], onDelete: Cascade)
  date      DateTime  @db.Date
  shiftType ShiftType
  workers   ShiftWorker[]
  @@unique([rosterId, date, shiftType])             // ← one row per calendar slot of the month
  @@index([rosterId, date])                         // calendar-grid reads
  @@map("shifts")
}

model ShiftWorker {                                 // table: shift_workers (junction)
  shiftId  Int
  shift    Shift  @relation(fields: [shiftId], references: [id], onDelete: Cascade)
  workerId Int
  worker   Worker @relation(fields: [workerId], references: [id], onDelete: Restrict)
  role     Role                                     // ← snapshot of worker.role at assignment time
  @@id([shiftId, workerId])                         // ← same worker never twice in one slot;
                                                    //   8 h each: cost = count × 8 × rate
  @@index([workerId])                               // per-worker schedule + daily-cap checks
  @@map("shift_workers")
}

model Alert {                                       // table: alerts
  id             Int       @id @default(autoincrement())
  rosterId       Int
  roster         Roster    @relation(fields: [rosterId], references: [id], onDelete: Cascade)
  type           AlertType
  detail         Json      @db.JsonB                // {date,shift,role} or {workerId,deficitHours}
  acknowledged   Boolean   @default(false)
  acknowledgedAt DateTime?
  @@index([rosterId])
  @@map("alerts")
}

Constraint-to-rule map: companies lower(name) UNIQUE → no duplicate employers, and the CSV resolve-or-create key; workers.national_id UNIQUE → no duplicate IDs and the CSV upsert key; contracts.worker_id UNIQUE → exactly one contract per worker; staffing_requirements (role, shift) UNIQUE → a single headcount setting per role × shift; rosters.month UNIQUE → one roster per month; shifts (roster_id, date, shift_type) UNIQUE → exactly one row per calendar slot; shift_workers PK (shift_id, worker_id) → the same worker can never occupy one slot twice, even under concurrent edits — because a shift row is a roster + date + shift type, this one key replaces the old flat (roster, worker, date, shift) uniqueness. The 2-shifts-per-day cap cannot be expressed by the database alone; it — and every other cross-row rule — is enforced by the RosterValidator on every write path.

Design decision: the shifts–workers connection is normalized into a shifts table (one row per roster + date + shift type, created at generation) plus a shift_workers junction, replacing the flat shift_assignments table. The junction snapshots the worker's role at assignment time, so a historical roster stays accurate even if the worker is later promoted or reassigned.

Schema optimization notes

  • Native Postgres enums for role, shift_type, worker and roster status, and alert type — compact storage, self-documenting, and invalid values are rejected by the database itself.
  • Every FK used in joins is indexed: workers.company_id, shifts.roster_id (leading column of the [rosterId, date] index), shift_workers.worker_id, and alerts.roster_id; shift_workers.shift_id is covered as the leading column of the composite PK.
  • Partial index ON workers (id) WHERE status = 'ACTIVE' (raw-SQL migration) — every engine and candidate query filters on active workers.
  • Cascade rules: ON DELETE CASCADE down roster → shifts → shift_workers and roster → alerts, so regenerating a draft month wipes cleanly in one delete; ON DELETE RESTRICT from companies → workers and workers → shift_workers, so no employer or shift history is ever orphaned.
  • No stored derived values: hours and cost are always computed as count × 8 × rate at read time, so totals can never drift from the assignments.
  • updated_at is maintained by Prisma's @updatedAt on mutable tables — no database triggers to migrate or debug.

pg-boss schema: pg-boss creates and migrates its own pgboss schema (pgboss.job, pgboss.archive, pgboss.schedule, …) on first start; it lives outside Prisma and needs no hand-written job table.

Design decision: the 21-boolean availability matrix is stored as one jsonb column (shape enforced by Zod on every write) rather than 21 columns — the engine always reads a contract whole and never queries by a single availability cell, and the CSV layer flattens/unflattens it at the boundary.

Core Algorithms & Validation

Israeli ID (Teudat Zehut) checksum — validators/israeliId.ts

function isValidIsraeliId(raw: string): boolean {
  if (!/^\d{1,9}$/.test(raw)) return false;
  const id = raw.padStart(9, '0');               // short IDs are left-padded with zeros
  let sum = 0;
  for (let i = 0; i < 9; i++) {
    const product = Number(id[i]) * (i % 2 === 0 ? 1 : 2);   // alternate weights 1,2,1,2…
    sum += product > 9 ? product - 9 : product;  // sum the digits of two-digit products
  }
  return sum % 10 === 0;
}

CP-SAT roster model — solver/solve_roster.py

# problem JSON on stdin → solution JSON on stdout; SHIFT_HOURS = 8
from ortools.sat.python import cp_model

def solve(p):                                     # p: month, workers+contracts, requirements
    m = cp_model.CpModel()

    # Decision vars x[w,d,s] — created ONLY where the contract allows (prunes the model)
    x = {(w.id, d, s): m.new_bool_var(f"x_{w.id}_{d}_{s}")
         for w in p.workers for (d, s) in p.slots
         if w.availability[weekday(d)][s]}

    # HARD (a): role coverage per day × shift × role, with shortfall slack
    short = {}
    for (d, s, role, required) in p.requirements:
        short[d, s, role] = m.new_int_var(0, required, f"short_{d}_{s}_{role}")
        m.add(sum(x[w.id, d, s] for w in p.workers
                  if w.role == role and (w.id, d, s) in x)
              + short[d, s, role] == required)

    # HARD (b): ≤ 2 shifts per worker per calendar day
    for w in p.workers:
        for d in p.days:
            m.add(sum(x[w.id, d, s] for s in "ABC" if (w.id, d, s) in x) <= 2)

    # HARD (c): max monthly hours; SOFT: min-hours shortfall slack
    deficit, load = {}, {}
    for w in p.workers:
        total = sum(v for (wid, _, _), v in x.items() if wid == w.id)
        m.add(8 * total <= w.max_monthly_hours)
        deficit[w.id] = m.new_int_var(0, w.min_monthly_hours, f"deficit_{w.id}")
        m.add(8 * total + deficit[w.id] >= w.min_monthly_hours)
        load[w.id] = scaled_load(m, total, w)     # 8·total·100 / contracted hours
                                                  # (add_division_equality)

    # Fairness: minimize spread between most- and least-loaded workers
    load_max = m.new_int_var(0, 100, "load_max"); m.add_max_equality(load_max, load.values())
    load_min = m.new_int_var(0, 100, "load_min"); m.add_min_equality(load_min, load.values())

    # Lexicographic objective via weights: coverage ≫ min-hours ≫ even distribution
    m.minimize(10_000 * sum(short.values())
             +    100 * sum(deficit.values())
             +      1 * (load_max - load_min))

    solver = cp_model.CpSolver()
    solver.parameters.random_seed = 42            # fixed seed        ┐ deterministic:
    solver.parameters.num_search_workers = 1      # single-threaded   ┘ same input → same roster
    solver.parameters.max_time_in_seconds = 30    # best feasible solution accepted at the limit
    solver.solve(m)                               # never INFEASIBLE — slacks absorb any shortage

    return {
      "assignments": [{"workerId": w, "date": d, "shift": s}
                      for (w, d, s), v in x.items() if solver.value(v) == 1],
      "alerts":
        [{"type": "unfillable_slot", "date": d, "shift": s, "role": r,
          "missing": solver.value(v)} for (d, s, r), v in short.items() if solver.value(v) > 0]
      + [{"type": "min_hours_shortfall", "workerId": w,
          "deficitHours": solver.value(v)} for w, v in deficit.items() if solver.value(v) > 0],
    }

RosterValidator rules — engine/validator.ts

One rule list in TypeScript, mirroring the solver's constraints and consumed by every manual-edit endpoint — manual edits go through this validator, never the solver. HARD failures return 422 and are never persisted; SOFT failures return 409 warnings that require ?confirm=true.

type Verdict =
  | { ok: true;  warnings: SoftWarning[] }     // persist; warnings gate on confirm=true
  | { ok: false; violations: HardViolation[] } // reject with 422 — no override exists

  maxTwoShiftsPerDay,        // would be the worker's 3rd shift on that calendar date

Full-sync semantics: each row is validated in full (Zod + ID checksum); company_name is resolved to an existing company by case-insensitive name or a new company is created; then, if national_id exists the worker and its contract are updated, otherwise both are created — in one transaction per row, so a failing row rolls back only itself and the batch continues. After all rows are processed, a sync sweep sets every existing worker whose national_id appears nowhere in the file to Inactive — the CSV is the authoritative workforce list. Deactivation never deletes: the worker keeps their contract, share token, and shift history, and is simply excluded from future rostering. A worker whose row is present but failed validation is not deactivated (they are in the file; only the bad row is reported). The import job's result is a per-row report:

type ImportResult = {
  totalRows: number; inserted: number; updated: number; failed: number;
  deactivated: number;                     // sync sweep: workers absent from the file
  deactivatedWorkers: Array<{ workerId: number; nationalId: string; name: string }>;
  errors: Array<{ row: number;            // 1-based data-row number
                  nationalId?: string;
                  field?: string;          // offending column, when attributable
                  message: string }>;
};
Design decision: import = sync. Because an accidental partial file would silently deactivate everyone omitted, the UI's import panel states this rule up front and requires an explicit confirmation ("workers not in this file will be deactivated") before upload; the completion report lists every deactivated worker by name so the action is visible and reversible (re-import a full file, or flip the worker back to Active in the UI).

pg-boss specifics

// jobs/queue.ts — shared by API (send) and worker.ts (work)
const boss = new PgBoss(process.env.DATABASE_URL); // owns the `pgboss` schema

export const QUEUES = { CSV_IMPORT: 'csv-import', ROSTER_GENERATION: 'roster-generation' };

// worker.ts
await boss.work(QUEUES.CSV_IMPORT,        { teamSize: 1 }, handleCsvImport);
await boss.work(QUEUES.ROSTER_GENERATION, { teamSize: 1 }, handleRosterGeneration);
// handleRosterGeneration: build problem JSON (engine/problem.ts) →
// spawn `python3 solver/solve_roster.py`, problem JSON on stdin, solution JSON on stdout →
// persist draft roster (shifts + shift_workers rows) + alerts in ONE transaction

// API: enqueue generation — singletonKey = month ⇒ at most ONE queued/active
// generation job per month; a second request gets null ⇒ respond 409
const jobId = await boss.send(QUEUES.ROSTER_GENERATION, { month }, { singletonKey: month });

// cron: 06:00 on the 25th of every month → auto-generate NEXT month's draft
await boss.schedule(QUEUES.ROSTER_GENERATION, '0 6 25 * *', { month: 'next' });

Job polling endpoint shape — GET /api/jobs/:id, read from pgboss.job; the SPA polls every 1.5 s until a terminal state:

{ "id": "8f1c…", "name": "csv-import",
  "state": "created" | "active" | "completed" | "failed",
  "createdAt": "…", "completedAt": "…" | null,
  "result": ImportResult                      // csv-import
          | { "rosterId": 42, "alertCount": 3 }  // roster-generation
          | { "error": "…" } }                // failed
Design decision: both job handlers run with retryLimit: 2 (exponential backoff) because they are idempotent — import upserts by national_id (and resolves-or-creates companies by name; the sync sweep re-deactivates the same absent set), and generation deletes-and-rewrites the month's draft shifts, shift-worker rows, and alerts inside a single transaction, so a retried job converges on the same state.
